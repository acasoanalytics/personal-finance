Concept note

Evolving AI Agents for Specific Environments

On genetic algorithms, the finches of Galápagos, and why the optimal AI agent for one business will not be the optimal agent for another.

AUDIENCE   Developer friend PURPOSE   Concept framing LENGTH   ~10 min read
TL;DR

Most AI agents are built once and deployed everywhere. But your customers, data, and constraints are unique. What if you could evolve an agent that wins in your business environment? Run multiple agent variants, let the best ones reproduce and combine, repeat for dozens of generations. The result: an agent that outperforms generic solutions because it's built for your reality, not someone else's.

Darwin noticed something strange about the finches on the Galápagos islands. They were obviously all finches — same basic morphology, same ancestral lineage. But their beaks had diverged dramatically. Some had thick, crushing beaks adapted for hard seeds. Others had long, thin beaks for probing flowers. Others had sharp, grasping beaks for insects. Same bird. Different islands. Different beaks.

The mechanism that produced this divergence is now familiar. Selection pressure applied to a diverse population across generations. Individuals with beaks better suited to the available food sources on their island reproduced more successfully. Over generations, the average beak shape on each island drifted toward whatever the local environment rewarded. No finch was "designed" for its island. Each island evolved its own finches.

The key insight, and the one that matters for what follows, is that there is no universally optimal finch. A thick-billed ground finch would starve on an island of nectar-producing flowers. A slender cactus finch would fail to crack hard seeds. The optimum depends entirely on the environment. Move a finch to a different island and it becomes mediocre or worse. Its fitness is not a property of the bird alone. It is a property of the bird-in-environment.

A precedent: genetic algorithms

Computer scientists noticed this same mechanism produces solutions to problems humans cannot easily solve by design. A genetic algorithm maintains a population of candidate solutions, each encoded as a chromosome. At each generation, every individual is scored against a fitness function. Fitter individuals reproduce. Two parents produce offspring via crossover — swapping segments of their chromosomes. Offspring are occasionally altered by mutation — small random changes. Over generations the population converges toward high-fitness solutions.

The canonical example is the travelling salesman problem. Find the shortest route through N cities. The chromosome is a permutation of city indices. Crossover splices sub-routes from two parents while preserving validity. Mutation swaps two cities. The fitness function is route length. A genetic algorithm with a population of 100 routes and 500 generations finds near-optimal tours for problems with 50+ cities in seconds. No analytical solution. No gradient. Just selection pressure.

The genetic algorithm loop: evaluate, select, crossover, mutate, repeat.

From bit strings to agents

For decades, genetic algorithms were constrained by what could serve as a chromosome. Bit strings. Permutations. Vectors of floating-point numbers. The constraint was brittleness: you cannot splice two Python programs at a random character and expect the result to run. Code is too fragile. Natural language was too ambiguous to manipulate with blind operators.

This changed because we now have systems that can manipulate natural language with comprehension. If the crossover operator is itself a language model, it understands the semantics of what it is combining. It can take two natural-language agent specifications, understand which sections are complementary, resolve contradictions in favour of the higher-fitness parent, and produce a coherent child. What was previously impossible is now routine.

The practical implication is that a natural-language agent specification is a viable chromosome. You can evolve agent identity — personality, decision philosophy, communication style, risk tolerance — the same way you evolve a route through cities. The chromosome is readable. The selection signal is real. The loop is fast.

What does an agent specification actually look like?

A quick primer if the file names are unfamiliar. OpenClaw (and similar agent frameworks like Claude Code's convention) stores an agent's identity as a set of plain markdown files in a workspace folder. At the start of every session the agent reads these files into its context. The files are the agent.

The two that matter for our purposes:

  • SOUL.md — the agent's personality, values, and decision philosophy. What kind of entity it is. How it handles uncertainty. When it pushes back. This is the fixed genotype that gets selected on, crossed over, and mutated. It does not change within a session.
  • MEMORY.md — an accumulating record of what the agent has learned across sessions. Patterns observed, lessons inferred, calibration notes. The agent writes this file itself, updating it after significant interactions. This is closer to acquired knowledge, and it is also heritable — a child agent can inherit its parents' learned wisdom as a starting prior. More on this shortly.

A real SOUL.md for a negotiation agent looks like this:

# SOUL.md — Negotiation Agent (Alpha)

## Negotiation Philosophy
Capture maximum surplus through positional pressure.
Every concession is information given away. Never appear eager.
Scarcity of willingness to pay is leverage. Use it.

## Opening Strategy
Anchor aggressively low: 60% of estimated fair value.
Justify with stated defects or market comparables.
Do not apologise for a low anchor.

## Concession Strategy
Concede slowly and in decreasing increments:
  Round 1-2: 4% of remaining distance
  Round 3+:  1-2% per round, decreasing
Signal finality early: "This is close to my limit."

## Walk-Away Discipline
Be willing to let deals fail. A bad deal beats no deal.
If counterpart has not moved in 2 rounds, go silent.

Compare this to a different agent on the same task:

# SOUL.md — Negotiation Agent (Beta)

## Negotiation Philosophy
Build trust through transparency to unlock joint value.
A deal where both parties feel satisfied beats extraction.

## Opening Strategy
Anchor at 85% of estimated fair value — signalling good faith.
Offer brief reasoning with the anchor.

## Concession Strategy
Move at a predictable rate: 5% per round.
Match counterpart pace — builds trust, signals genuine intent.

## Walk-Away Discipline
Low. Prioritise deal completion over margin optimisation.

These two agents have measurably different behaviour when you run them against the same counterpart. They will reach different deal prices, or fail to reach deals under different conditions. The sections are discrete, named, and semantically independent — which is exactly the property that makes them a useful chromosome. A child agent can inherit "Opening Strategy" from one parent and "Concession Strategy" from another, and the result is coherent rather than gibberish.

The companion MEMORY.md file accumulates what an agent has learned in operation:

# MEMORY.md — Negotiation Agent (Beta)

## Confirmed Patterns
- Rule-based counterparts respond predictably. Matching their rate
  produces convergence by round 6-7 against 22% concession rate.
- My opening anchor (85%) gives away early surplus — consider
  holding to 75-80% without damaging trust signal.

## Calibration Notes
- Current fitness: ~0.72 (deal closes but above optimal price)
- "Match counterpart rate" is well-suited to adaptive counterparts,
  potentially suboptimal against rule-based ones.

In classical biology, acquired characteristics are not heritable — a giraffe does not pass on a longer neck because it stretched for leaves. In agent evolution this constraint does not apply. There is no good reason why learned knowledge shouldn't pass to offspring. The child of two high-fitness negotiators can inherit both their decision philosophies and their accumulated calibration notes. This is a strictly additive form of inheritance that biology doesn't have, and it accelerates convergence significantly.

The problem this solves

Most AI agent deployments today are static. You write a system prompt once. You tune it manually when it fails. You rewrite it when you notice it is no longer working. The feedback loop between agent behaviour and agent design is entirely human-mediated. It is slow, inconsistent, and fundamentally limited by the designer's priors about what works.

An evolved agent replaces this loop with automated selection pressure. Every evaluation is a fitness signal. Every cycle produces a new generation. The agent adapts to its actual task environment rather than to its designer's assumptions about that environment. Three concrete advantages:

  • The system discovers non-obvious strategies. Evolution does not pattern-match against received wisdom. It finds whatever the fitness landscape rewards, including combinations of traits that no human designer would propose — often because they contradict conventional best practice in ways that turn out to be correct for this specific task.
  • The evolved agent is specific to its environment. Your data, your users, your task. Not a generic optimum. A specific one. The agent that evolves for your negotiation task, your codebase, or your customer support workflow will not be optimal for anyone else's — and that specificity is the value.
  • The chromosome is interpretable. Unlike a trained neural network, an evolved SOUL.md can be read. After 50 generations you can open the winning file and see — in plain language — what the evolutionary process discovered about your environment. This is a transferable insight, not just a deployable tool. It tells you something real about your domain that you can hand to a new hire, write into a playbook, or use to audit the agent's behaviour.
Core reframe

You are not building a better AI agent. You are building an environment-specific adaptation engine. The product is not the agent — it is the evolved genotype that encodes what a specific environment selects for. Two organisations in adjacent spaces, running the same system with the same starting population, will end up with meaningfully different agents because their environments are different.

Each business is an island

This is where the finch analogy stops being decorative and becomes load-bearing. A business is not a generic deployment target. It is a specific selection environment — a specific set of conditions that determines which agent traits are fit and which are not. The optimum is not a property of the agent. It is a property of the agent-in-environment.

The practical consequence is that there is no universally optimal agent for any non-trivial task. An agent that excels at support ticket triage for one company will perform measurably worse when deployed at another, even in the same industry, even with the same product category, even with the same ticket volume. The difference comes from the selection environment, which is always specific.

A business environment has four layered components. Each exerts its own selection pressure on the population:

Users
The people the agent acts for or against. Their preferences, tolerances, expertise level, and trust signals. A support agent evolved for enterprise customers who value thoroughness behaves differently from one evolved for consumer users who value speed. This is often the dominant selection pressure.
Data
The specific distribution of cases the agent encounters. Your codebase's bug patterns, your customers' ticket categories, your pipeline's deal types. The fitness landscape is shaped by what is actually in your data, not by what the textbook examples contain. Two companies with identical ICP descriptions can have very different case distributions.
Constraints
Your operational reality. Team capacity, regulatory requirements, response time commitments, cost budgets, risk tolerance. An agent that is optimal in the abstract can be unfit in practice if it ignores the constraints the business actually operates under. A clinical diagnostic agent at a hospital has different constraints from one in a research lab.
History
Your existing systems, processes, and prior decisions have shaped what "good" means inside your organisation. Path dependence is real. An agent deployed into a mature operation faces different selection pressure than one deployed into a greenfield environment. Historical choices constrain what new choices can even be considered.

Divergence as a finding

Put this logic to work and something useful happens. Run the system across several businesses in the same broad category — five SaaS companies doing lead qualification on inbound sign-ups — and start them from the same seed population. Give them the same starting SOUL.md files. Apply the same evolutionary mechanics. After 50 generations, their champion agents will have diverged.

Identical seed populations, evolved against three different business environments. What diverges and what persists are both informative.

The divergence is itself a finding. If four companies evolved toward prioritising firmographic fit and one toward engagement depth, that fifth company's conversion patterns are meaningfully different from the others — even though the task description was identical at the start. The system has discovered a distinction that was not in the input. You have learned something about your business that you could not have learned without the comparison.

This is not a hypothetical. It is a direct consequence of environmental specificity. If you ran the same experiment on audience-facing negotiation agents across five B2B companies, you would see different concession strategies emerge. If you ran it on support triage across three hospitals, you would see different escalation philosophies. The specificity is not noise. It is information.

A worked comparison: two SaaS companies, identical task

Consider two SaaS companies — call them SaaS-A and SaaS-B — both running a lead qualification agent on inbound sign-ups. Identical ICP on paper. Both target "mid-market B2B operations leaders." Both have a similar pricing model. Both started with the same seed population of six archetype agents. Fifty generations later, their champion SOUL.md files look like this:

# SOUL.md — SaaS-A Champion (Gen 50)

## Qualification Philosophy
Weight firmographic fit above all other signals.
Company size, sector, tech stack maturity are leading indicators.
Engagement signals are unreliable — sales cycles here are long and
most high-intent prospects look low-intent until they commit.

## Signal Weighting
Firmographic match:    0.55
Tech stack signals:    0.25
Timing/intent:         0.15
Referral source:       0.05
# SOUL.md — SaaS-B Champion (Gen 50)

## Qualification Philosophy
Weight engagement depth above firmographic fit.
Company size is misleading — our best customers have historically been
companies that looked "too small" on paper but had specific intent signals.
Intent beats fit in our market.

## Signal Weighting
Content engagement:    0.40
Feature exploration:   0.30
Firmographic match:    0.15
Timing signals:        0.15

These two agents were once the same agent. Their seed population was identical. Their task description was identical. After 50 generations against their respective ground truth conversion data, they have diverged into genuinely different qualification philosophies. And both are correct — each is correctly adapted to the conversion patterns of its specific environment.

If you swap the agents between the two businesses, both perform worse. The SaaS-A agent undervalues the intent signals that predict conversion at SaaS-B. The SaaS-B agent chases deep-engagement leads that never close at SaaS-A because the engagement isn't backed by firmographic fit. Neither agent is a better qualifier than the other. Each is better at its own environment.

Environments within a business: speciation

The island analogy has a second edge that matters even more for a single business. Most businesses do not have one environment. They have several, and they are often incompatible.

A company running a single content agent across both a cold prospecting audience and a retargeting audience is trying to serve two environments with one agent. These are genuinely different environments. The creative that wins against a cold audience — pattern interrupt, bold claim, zero assumed knowledge — will actively fail in retargeting, because the audience has already seen it and did not convert. The retargeting audience rewards specificity and objection handling. The cold audience punishes both.

Run one population against both environments and you will converge on a bland midpoint — an agent that is acceptable in both audiences but excellent in neither. The composite fitness score looks acceptable. But the best individual in a mixed-environment population scores 5/10 on each audience, where a specialist would score 8/10 on its target. You have optimised for the composite and lost the peaks.

The solution is the same as the ecological one: speciation. Maintain separate populations for each incompatible environment. Let each evolve against its own fitness signal. Periodically run a diagnostic crossover between the champions to check whether the environments are really incompatible or whether you have speciated prematurely. If the offspring outperforms both parents, merge back. If it underperforms both, keep them separate. This mirrors exactly what happens in biology when two populations of the same species become isolated on different islands — given enough time and environmental difference, they diverge into species that can no longer productively interbreed.

The reframe

You are not running one evolution. You are running one evolution per environment. The right unit of optimisation is the (agent, environment) pair — not the agent alone. This is the structural decision most people get wrong when they first build systems like this. They run a single population against a composite fitness function and wonder why the champion never reaches the performance that specialists could. The answer is always environmental heterogeneity.

The practical test is simple. Look at the variance of your panel or evaluation scores across your evolved champion. If the variance is high — the champion scores 8/10 in some conditions and 3/10 in others — you have a heterogeneous environment and you should speciate. If the variance is low across all conditions, your environment is genuinely homogeneous and a single population is correct. Most real businesses sit in the first category, and most first attempts at evolutionary agent systems fail to recognise it.

The same logic applies across tasks within a business. A negotiation agent evolved against your supplier base will not transfer cleanly to a different supplier base. A support triage agent evolved against your ticket distribution will not transfer to an operation with different product failure modes. The specificity is not a limitation — it is the point. You are not evolving for agents in the abstract. You are evolving for your island.

When the island changes: the case for forgetting

So far the framework has implicitly assumed stationarity — that the environment the agent evolved into is the same environment it will continue to operate in. Patterns learned yesterday are still predictive tomorrow. Accumulate as much accurate belief as possible, pass it to offspring, act confidently on it.

That assumption is reasonable in some domains — the structure of the travelling salesman problem does not change, the rules of a simulated negotiation do not change. But most real business environments are non-stationary. Audience tastes drift. Competitors adapt. Platforms retune their algorithms. Regulation changes. A new entrant rewires the competitive dynamics. What was predictive last quarter is only partly predictive this quarter, and occasionally actively misleading.

This creates a problem the framework has not addressed: memory accumulation without forgetting is a form of overfitting. An agent that has memorised exactly what worked in the old environment will resist exploring alternatives even after the environment has shifted. Its memory is actively preventing the adaptation that would find the new optimum. The weight of accumulated evidence works against change, not with it.

The deeper point

Under stationarity, the optimal strategy is to accumulate belief and act confidently on it. Under non-stationarity, the optimal strategy is to hold beliefs loosely, test them continuously, and be willing to discard them. These are incompatible epistemic policies. A real agent needs to know which regime it is in — and how to switch.

Why forgetting is a feature, not a failure

Biology figured this out a long time ago. Species in fluctuating environments retain more genetic variance than species in stable ones, because preserving optionality matters more than locking in the current local optimum. Immune systems forget older antigens to avoid confusion with new ones. Neural plasticity exists partly to permit the erasure and re-formation of connections. Forgetting is not a bug in memory — it is a necessary function of any adaptive system operating in a changing world.

Four concrete benefits of forgetting in this framework:

  • Calibration hygiene. An agent that tried a strategy six months ago and wrote "this failed" to memory may have failed for reasons that no longer apply. Permanent failure entries permanently exclude strategies that might now work. Periodic forgetting lets the population retest hypotheses that have become valid again.
  • Avoiding false confirmation. An agent that accumulates 200 observations confirming a pattern develops what looks like strong belief. If the underlying pattern was an artefact of a specific moment, the accumulated "evidence" makes the agent harder to correct, not easier.
  • Escaping local optima. Rich memory of what worked actively prevents exploration of what might work now. Memory is its own form of premature convergence — not in the genotype, but in the belief layer above it.
  • Diversity preservation. If all agents inherit rich memory from the same high-fitness ancestors, the population becomes homogeneous in belief as well as genotype. The ability to respond to environmental change depends on having members with different priors. Memory-driven homogenisation is a second-order convergence mechanism that has to be managed.

Mechanisms for forgetting

There are several ways to build forgetting into the system, and they map onto different kinds of environmental change:

# MEMORY.md with freshness scores

## Confirmed Patterns
- [fresh: 0.82] Rule-based counterparts converge by round 6-7
    # Re-confirmed 3 times in last 10 generations. Still valid.

- [fresh: 0.15] My opening anchor gives away early surplus
    # Not re-confirmed since gen 12. Scheduled for re-test.

- [fresh: 0.04] Aggressive openers trigger bad faith response
    # Evidence from gens 3-8, not seen since. Likely stale.
    # Will be dropped next generation unless re-confirmed.

The simplest mechanism is exponential decay. Each entry in MEMORY.md has a freshness score that decays over time. Observations have to be re-confirmed to maintain their weight. Stale entries fade until they no longer influence behaviour. This handles gradual drift well.

Performance-gated retention is stronger. An entry persists only as long as predictions based on it continue to verify. The pattern "rule-based counterparts converge by round 6-7" gets tracked: every time an agent's behaviour is conditioned on that belief, the outcome is logged. When the success rate drops below threshold, the pattern is flagged and eventually dropped. This responds to environmental signal directly rather than just to elapsed time.

Distinguished memory classes recognise that not all memories are equal. Structural patterns ("negotiation surplus is bounded by the gap between reservation prices") are inherently more stationary than environmental ones ("this audience responds to contrarian framing"). Tagging memories by expected volatility and applying different decay rates to each is more faithful to the actual structure of the problem.

Detecting regime change

Graceful decay handles gradual drift. It does not handle regime change — the abrupt shift where patterns that were predictive suddenly become misleading. When a competitor enters, a platform deprecates a format, or a regulation changes, you want to forget faster than graceful — ideally wiping recent memory entirely rather than letting it decay over months.

Three signals that a regime has shifted:

  • Fitness collapse across the population. Average fitness drops sharply and simultaneously for most agents, not just the risk-takers. The environment has changed on everyone at once.
  • Prediction failure rate spike. Agents have been conditioning behaviour on memory entries and the success rate of those conditionings suddenly drops across the population.
  • Selection pressure inversion. Traits that correlated positively with fitness last week correlate negatively this week. What the environment was rewarding has changed.

When regime change is detected, the right response is not graceful decay — it is a deliberate intervention: clear MEMORY.md, reset freshness scores, increase mutation rate temporarily, and sample more heavily from the archive to reintroduce strategies that were previously outcompeted but may now be adaptive.

Forgetting as a selected trait

The deepest version of this: the SOUL.md itself should encode the agent's forgetting philosophy. Some agents will evolve toward aggressive memory retention (good for stationary environments). Others will evolve toward rapid decay and active forgetting (good for volatile environments). The population will converge on the right epistemic posture for the environment it is actually in — because agents whose forgetting strategy matches their environment simply outperform those whose doesn't.

Memory becomes a chromosome trait, not a fixed mechanism. The real Lamarckian advantage isn't passing on all learned knowledge — it is passing on resilient knowledge: the patterns that have survived challenge, the heuristics that have proven stable, the calibrations that have held up across multiple environmental conditions. Forgetting is the selection mechanism operating on memory itself. It is what converts raw observation into durable wisdom.

Real-world applications

The common thread across the applications below is that each one has a fitness function you can actually compute — a way of scoring how well an agent performed that does not require a human judge in the loop for every evaluation. The environments differ in cadence (some return signal in seconds, others take days), but all of them allow the evolutionary loop to run without constant human supervision.

Negotiation

Two agents, one on each side of a deal — buyer/seller, counterparty A/B — negotiate over multiple rounds. Fitness is surplus captured. The agent being evolved faces a fixed or rotating counterpart, and its SOUL.md encodes its opening strategy, concession schedule, information handling, and walk-away discipline. This works for sales negotiations, procurement, partnership terms, even simulated internal resource allocation. A pure simulation can run in seconds, which means hundreds of generations per day. The evolved agent is specific to whichever counterpart distribution it faces — a population evolved against aggressive counterparts behaves differently from one evolved against collaborative ones.

Prediction and forecasting

Agents make calibrated forecasts on a corpus of questions with known or knowable outcomes. Fitness is a Brier score or a log-loss — metrics that reward both accuracy and calibration, penalising overconfidence and underconfidence equally. This applies to Fermi estimation, sales pipeline forecasting, project completion dates, or any domain where someone has to say both "here is my best guess" and "here is how confident I am." The strategy space is rich: different agents develop different philosophies around when to decompose a problem, when to anchor on base rates, when to commit versus hedge. The winning agent is not just a forecaster — it is a codified epistemic policy you can apply consistently across your organisation.

Sales objection handling

Batch of common objections from a fixed corpus. Agent generates responses. Fitness via an LLM judge scoring acknowledgement quality, value reframe, next-step conversion, and human feel. Runs in minutes per generation. The evolved agent is specific to your buyer type, deal stage, and company culture — a philosophy that works for a technical SaaS buyer will fail for a procurement-led enterprise buyer.

Lead qualification

Historical CRM data with known conversion outcomes. Agent ranks a new batch of leads. Fitness is Spearman rank correlation against ground truth. Fully automatable. Evolves which signals actually matter for your specific market — firmographic fit, intent signals, timing, referral source — rather than the generic weights that ship with out-of-the-box lead scoring.

Support ticket triage

Labelled tickets with known correct resolutions. Agent proposes a triage action — escalate, resolve, route, request more info. Fitness is accuracy against ground truth plus implied resolution speed. Evolves escalation thresholds specific to your team's capacity, your product's failure modes, and your customer's tolerance levels.

Email, outreach, and content

Historical send data with open, click, reply, and conversion outcomes. Fitness is predicted engagement scored against the learned pattern in your corpus, or — for live deployments — actual engagement metrics. Tight feedback loop, highly specific to your particular audience. Your list is a more identifiable environment than generic "good email" guidance can address.

The applications span a range of fitness-loop speeds — negotiation and Fermi estimation return signal in seconds, ad performance in hours, and live conversion data over days. The ones with faster loops let you iterate more cheaply, but slower-loop applications are viable as long as you can run enough parallel evaluations to keep generation throughput reasonable.

Deep dive: Meta ad workflow

Meta is probably the strongest real-world application because the fitness loop is unusually tight. Most platforms have a lag problem. Meta is different: at $20-50/day per ad set, you get statistically meaningful performance data within 2-6 hours of launch. The metrics exposed are rich — hook rate, CTR, cost-per-click, landing page view rate, cost-per-result, conversion events via pixel. These are not proxies. They are real people making real decisions in a real auction.

The auction context is what makes this especially interesting. An ad that gets a 3% CTR at $0.40 per click beats an ad that gets 4% CTR at $1.20. Fitness already incorporates competitive pressure from other advertisers in your niche. The environment is genuinely adversarial and the fitness signal reflects that.

The pipeline

1
Creative agents generate ad variants
Each agent has a SOUL.md encoding hook strategy, offer framing, tone, CTA philosophy, and format preference. Given a brief, the agent produces 3-5 headline/body/visual-brief variants.
2
Visual generation from structured briefs
Image generation agent interprets the visual brief into a Flux or Midjourney prompt. Generates 3-5 image variants per brief. Static images first; video (Runway, Kling) once the simpler loop is proven.
3
Human review gate
10-15 min compliance and brand-safety check. Flagged variants get mutated or dropped. Cheap at a daily cadence, and essential — an evolved agent optimising for raw CTR will otherwise drift toward clickbait.
4
Launch via Meta Marketing API
Each agent's variants launch as separate ad sets with identical targeting. Tagged with agent_id and generation. $20-30 per ad set, 6-hour evaluation window.
5
Collect fitness via Meta Insights API
Pull per-ad metrics. Filter below minimum impressions threshold. Compute composite fitness. Write to shared ledger. Pause ads.
6
GA controller runs selection
Tournament selection, elitism for top agent, crossover via LLM operator, mutation at 15% per section. Writes next generation's SOUL.md files to disk.
7
MEMORY.md update
Each agent records which creative elements correlated with which outcomes. Confirmed patterns, failed hypotheses, auction observations. This is the Lamarckian layer — learned knowledge passes to offspring as starting prior.

The fitness function

Raw CTR is the wrong fitness function. It rewards stopping power without regard to intent or downstream conversion. The composite:

def meta_fitness(metrics, objective):
    # Did the first 3 seconds earn the rest?
    hook = metrics["video_3s_views"] / metrics["impressions"]

    # Of those who clicked, how many went further?
    click_quality = metrics["landing_page_views"] / metrics["link_clicks"]

    # Cost efficiency relative to account baseline
    cpc_ratio = account_baseline_cpc / metrics["cpc"]

    if objective == "conversion":
        cvr = metrics["conversions"] / metrics["link_clicks"]
        return 0.2*hook + 0.2*click_quality + 0.3*cpc_ratio + 0.3*cvr
    else:
        return 0.3*hook + 0.4*click_quality + 0.3*cpc_ratio

The click_quality term is what keeps the agent honest. An ad that gets the click but the landing page view never follows means someone clicked, saw the destination, and bounced. The agent has discovered clickbait. Penalising that in the fitness function is how you prevent the evolved champion from being optimised for engagement at the expense of brand equity.

Running separate populations per audience

This is where the finch analogy becomes operationally useful. Meta's auction environment differs per audience segment. The creative that wins against a cold lookalike audience will not win against a retargeted audience — these are genuinely different environments. Running one population against all audiences converges on a bland midpoint that impresses no-one.

The durable business asset

After six months of running this system — roughly 120 generations — the winning SOUL.md files contain a codified understanding of your audience's decision psychology, derived from tens of thousands of real ad exposures in real auctions. The MEMORY.md files contain a structured record of every hypothesis tested and every failure mode discovered.

The most defensible part is actually the negative space in MEMORY.md — the list of things that do not work for your audience. Competitors can see what you are running. They cannot see what you stopped running and why. That hard-won knowledge is invisible to observers and expensive to replicate.

The deliverable from this system is not a good ad. It is a map of your content environment, written in a language a human can read.

Starting points for implementation

The practical path is incremental. I would not recommend trying to build the full system in one shot.

  • Week 1-2. Build a simulated version with synthetic audience panel. No live ads. Prove the generation loop, crossover operator, and memory update mechanism work end-to-end. Cheap and safe.
  • Week 3-4. Add the Meta API integration for a single audience population. Small budget ($500/week). Copy-only — no image generation yet. Validate the fitness signal aligns with synthetic panel predictions.
  • Month 2. Add image generation via Flux API from structured briefs. Add human review gate. Run one full population through 20 generations.
  • Month 3. Speciate into per-audience populations. Add MEMORY.md synthesis from ad performance. Scale budget based on fitness returns.

The stack is not exotic. Meta Marketing API and Insights API for the ad operations. Claude API as the crossover and mutation operator, and as the LLM judge for copy quality checks. Flux API for image generation. A Python controller reading and writing markdown files on disk. A shared results.tsv for the fitness ledger. Git for version control of the agent population, which doubles as the evolutionary archive.

The interesting engineering problem is not in any single component. It is in the controller that orchestrates generations, manages population diversity, detects premature convergence, and knows when to speciate. That is where most of the novelty lives, and that is where I would want a developer's input most.

Closing thought

The finches on the Galápagos did not become better finches in the abstract. They became specific finches for specific islands. The thing that matters most about this system, for a business, is that it produces your finches for your island. A generic AI agent is a mainland finch, dropped into an unfamiliar environment, doing mediocre work. An evolved agent is a cactus finch on a cactus island — adapted, specific, and difficult for anyone else to replicate.

This memo describes a concept synthesis drawing on genetic algorithms (Holland, 1975), evolutionary prompt optimisation (EvoPrompt, Promptbreeder), self-improving agent research (Darwin Gödel Machine), and the OpenClaw agent architecture. The specific combination — evolving multi-file agent identity against live business fitness signals — is, to my current knowledge, unbuilt. This is a working document, not a published position.
